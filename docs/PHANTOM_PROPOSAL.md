# PHANTOM™ — Авторская методика нейронной гуманизации текста

## Проблема: Замкнутый круг

```
Rule-based трансформации ──→ меняют слова/структуру ──→ не влияют на 
                                                        нейронные признаки
LLM-перефразирование ──→ создаёт новый текст ──→ нарушает оффлайн-стратегию
```

**Нужен третий путь** — собственная нейронная модель, которая:
- Работает 100% оффлайн (встроена в пакет)
- Весит 5-50 MB (не 7 GB как LLM)
- Не генерирует текст с нуля (это непосильная задача для маленькой модели)
- **Вместо этого**: целенаправленно модифицирует текст, ориентируясь на
  конкретные числовые признаки нашего же детектора

## Философия PHANTOM™

**PHANTOM** = **P**erceptual **H**umanization via **A**dversarial **N**eural
**T**ext **O**ptimization and **M**orphing

Ключевое отличие от всех существующих подходов:

| Подход | Что делает | Почему (не)работает |
|--------|-----------|---------------------|
| Rule-based | Заменяет слова по словарю | Не знает, КАК замена влияет на признаки |
| LLM | Генерирует новый текст | Знает язык, но не знает детектор |
| **PHANTOM™** | Оптимизирует текст **против известной модели** | Знает ТОЧНУЮ формулу детектора и бьёт по ней |

### Аналогия

Представьте, что детектор — это замок с 35 пинами. 
- Rule-based — это отмычка вслепую (может сломать 10 пинов, но не знает какие).
- LLM — это универсальная мастер-ключевая (работает, но стоит как сейф).
- **PHANTOM™ — это 3D-скан замка + CNC-фрезеровка точного ключа**.

Мы ВЛАДЕЕМ детектором. Мы знаем все 35 признаков, их веса, нормализацию.
Это **белый ящик**, и PHANTOM™ эксплуатирует это.

## Архитектура: 3 уровня

### Уровень 1: ORACLE (анализатор)
**Назначение**: Определить, КАКИЕ именно признаки тянут скор вверх и НА СКОЛЬКО.

```
Input text (AI-сгенерированный)
       │
       ▼
┌─────────────────────┐
│  Feature Extraction  │  ← 35 признаков (уже есть в neural_detector.py)
│  (pure Python)       │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│  GRADIENT ORACLE     │  ← Вычисляет ∂score/∂feature для каждого признака
│                      │     через MLP forward+backward (веса известны)
│  Output: ranked list │     Пример: "word_length_variance: сей час 11.3,
│  of feature gaps     │              вносит 0.34 в скор, нужно ≤ 6.0"
└──────────┬──────────┘
           │
           ▼
  Feature Gap Vector: [(f3, -5.3, 0.34), (f29, -1.25, 0.20), ...]
```

**Размер**: 0 MB (чистая математика поверх существующего MLP).
**Сложность**: Средняя (аналитические градиенты двуслойного MLP).

### Уровень 2: SURGEON (нейронный редактор)
**Назначение**: Принимать решения КАКИЕ слова/фразы заменить, НА ЧТО, и ГДЕ
изменить структуру — используя обученную нейронную модель.

Это **НЕ LLM**. Это 3 маленьких специализированных нейросети:

#### 2A: WordAlternator — Нейронный подборщик замен (5-10 MB)
```
Input:  (слово, контекст ±5 слов, текущая длина, целевая длина)
Output: топ-5 замен с оценкой (семантическое сходство × пользу для признака)
```

**Архитектура**: Word2Vec-подобная embedding (300 dim) + 2-layer FFN scorer.
- Embedding-матрица: 30K слов × 300 dim = ~36 MB (float16 → 18 MB; quantized int8 → 9 MB)
- FFN scorer: (300+4 inputs) → 128 → 64 → 1 = ~100 KB
- **Общий размер: ~10 MB** (quantized)

**Преимущество над SynonymDB**: WordAlternator учитывает КОНТЕКСТ.
Для "natural language processing" слово "natural" получит эмбеддинг,
специфичный для этого контекста, и НЕ будет заменено на "afro".

**Обучение**:
- Берём любой word2vec/fasttext на 1M+ текстов (есть готовые, Public Domain)
- Скоригруем scorer на задачу: "замена X→Y снижает признак fK на δ при сохранении косинусной близости ≥ 0.7"
- Данные создаём автоматически: берём 10K текстов, для каждого слова вычисляем, как его замена влияет на 35 признаков

#### 2B: SentenceReshaper — Нейронный переструктуризатор (3-8 MB)
```
Input:  (предложение, позиции слов, current_feature_gaps)
Output: позиция_разреза | тип_трансформации (split/merge/reorder)
```

**Архитектура**: Lightweight char-CNN (1D conv) + pointer network.
- CharCNN encoder: 3 conv layers (32→64→128 filters) = ~200 KB
- Pointer decoder: 128-dim attention + linear = ~50 KB
- **Общий размер: ~1 MB**

**Что делает**: Не генерирует новый текст. Предсказывает ПОЗИЦИЮ,
где нужно разрезать/соединить предложения, чтобы оптимизировать
`sentence_length_variance`, `burstiness_score`, `consec_len_diff_var`.

**Обучение**:
- Создаём 100K пар (sentence + optimal split point) автоматически:
  для каждого предложения перебираем все возможные точки разреза,
  вычисляем итоговый feature gap, берём лучшую как target.
- Loss = CrossEntropy(predicted_position, best_position)

#### 2C: FeatureChaser — Целевой оптимизатор (1-3 MB)
```
Input:  (current 35 features, target 35 features)
Output: стратегия = [(action, weight)]
        Пример: [("shorten_words", 0.8), ("split_sent", 0.4), 
                  ("inject_hedge", 0.3), ("remove_transition", 0.6)]
```

**Архитектура**: Simple 2-layer MLP: 70 → 64 → 32 → num_actions (~15).
- **Общий размер: <100 KB**

**Что делает**: Решает, КАКУЮ КОМБИНАЦИЮ действий применить и с какой 
интенсивностью. Обучен на 50K примеров (feature_gap → best_strategy).

### Уровень 3: FORGE (итеративный конвейер)

```
                    ┌──────────────────────┐
                    │   ORACLE (Level 1)   │
                    │   Feature gaps       │
    Input text ──→  └──────────┬───────────┘
                               │
                    ┌──────────▼───────────┐
                    │  FeatureChaser (2C)   │
                    │  → Strategy vector    │
                    └──────────┬───────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                 ▼
    ┌─────────────────┐ ┌───────────────┐ ┌───────────────┐
    │ WordAlternator   │ │ SentenceReshaper│ │  Rule-based    │
    │ (2A)            │ │ (2B)           │ │  (existing)    │
    │ Replace words   │ │ Restructure   │ │  ASH, burstiness│
    └────────┬────────┘ └──────┬────────┘ └──────┬────────┘
              │                │                  │
              ▼                ▼                  ▼
    ┌─────────────────────────────────────────────────────┐
    │                   MERGE                              │
    │   Apply changes, resolve conflicts, quality check    │
    └──────────────────────┬──────────────────────────────┘
                           │
                    ┌──────▼───────────┐
                    │   ORACLE again    │
                    │   Recompute gaps  │
                    └──────┬───────────┘
                           │
                     score < 0.34?
                      /          \
                    YES           NO → Loop (max 5 iterations)
                     │
                     ▼
               Output (humanized text)
```

**Ключевая новизна**: итеративный цикл с ГАРАНТИЕЙ улучшения на каждом шаге,
потому что ORACLE вычисляет точные градиенты. Это не слепой перебор —
это **градиентный спуск по пространству текстовых трансформаций**.

Нам НЕ нужны парафразы от людей. Всё можно сгенерировать автоматически:

#### Для WordAlternator:
1. Берём 10K текстов из любого корпуса (Wikipedia, CC-100, etc.)
2. Для каждого слова в каждом тексте:
   - Вычисляем 35 признаков без слова (замена на каждый из 5K кандидатов)

1. Берём 50K предложений

#### Для FeatureChaser:
1. Берём 10K AI-текстов (генерируем через GPT API одноразово, или из датасетов)
2. Применяем все комбинации действий, записываем (gaps_before, strategy, gaps_after)
3. Итого: ~50K примеров

## Ожидаемые результаты

### Консервативная оценка (без overfitting)

| Метрика | Сейчас (rule only) | С PHANTOM™ | Конкуренты (LLM) |
|---------|-------------------|------------|-------------------|
| Ср. detect_ai скор | 0.58-0.62 | **0.28-0.35** | 0.10-0.20 |
| Bypass rate (<0.40) | 0% | **60-80%** | 85-95% |
| Качество текста | Ошибки | Хорошее | Отличное |
| Время обработки | 1.5 сек | 2-4 сек | 5-15 сек |
| Размер | 7 MB | **17 MB** | 1.3-7 GB |
| Оффлайн | ✅ | ✅ | ❌ |

### Почему 60-80% bypass реалистичен

1. **ORACLE гарантирует**: каждая итерация уменьшает скор (градиентный спуск)
2. **WordAlternator**: замена с учётом контекста устранит текущую проблему
   (word_length_variance 11→5 без "language"→"albanian")
3. **SentenceReshaper**: нейронное разбиение vs текущее правильное (ломает грамматику)
4. **FeatureChaser**: вместо одной фиксированной стратегии — адаптивная
   комбинация действий под конкретный текст

### Почему НЕ 90%+

- WordAlternator с 30K словарём не может делать deep paraphrasing
  (не может "перестроить предложение другими словами")
- Некоторые тексты имеют настолько AI-шную семантическую структуру,
  что без полного перефразирования (LLM) их не исправить
- Наш детектор — один из нескольких. GPTZero/Originality используют свои модели,
  мы оптимизируем только под свою

## Альтернативы (рассмотренные и отвергнутые)

### A. Distilled T5-Small (60M params, ~240 MB)
- **Плюс**: Может генерировать текст, доказанное качество парафраз
- **Минус**: 240 MB — в 30x больше PHANTOM; нуждается в torch (~500 MB); нарушает zero-deps; качество на RU/UK низкое без fine-tune
- **Вердикт**: ❌ Нарушает архитектурный принцип zero-deps

### B. ONNX Runtime + мини-трансформер (50-100 MB)
- **Плюс**: Может быть хорошим паперфразером
- **Минус**: Требует onnxruntime (~50 MB pip пакет); 50-100 MB модель; инференс ~2 сек на CPU
- **Вердикт**: ⚠️ Возможно как опциональный `phantom[neural]` extra

### C. GPT-2 Small (124M params, ~500 MB)
- **Плюс**: Генеративная модель, может перефразировать
- **Минус**: 500 MB + torch; слишком большой; качество без fine-tune низкое
- **Вердикт**: ❌ Слишком тяжёлый

### D. Чисто rule-based (дальнейшее улучшение текущего подхода)
- **Плюс**: Ноль ML, ноль данных
- **Минус**: Потолок ~60-65% bypass, мы его уже достигли
- **Вердикт**: ❌ Потолок достигнут

## Почему PHANTOM™ — это радикально новый подход

### 1. Не существует аналогов
Все коммерческие решения используют LLM. Все open-source используют rule-based.
**PHANTOM™ — третий путь**: маленькие специализированные нейросети,
обученные ПРОТИВ ИЗВЕСТНОЙ ЦЕЛЕВОЙ ФУНКЦИИ.

### 2. Математическая гарантия улучшения
В отличие от rule-based (надеемся, что замена поможет) и LLM (надеемся,
что перефразировка обманет), PHANTOM™ вычисляет ТОЧНЫЙ ГРАДИЕНТ
и движется по нему. Каждая итерация гарантированно уменьшает скор.

### 3. Adversarial co-evolution
Если мы улучшим детектор (добавим признаки, обновим веса),
PHANTOM™ можно переобучить автоматически — данные для обучения
генерируются из самого детектора. Это **arms race встроенный в архитектуру**.

### 4. Принцип минимальности
PHANTOM™ не пытается быть LLM. Он решает ровно одну задачу:
"изменить текст МИНИМАЛЬНО, чтобы 35 чисел стали другими".
Для этой задачи 10 MB достаточно.

### 5. Zero external dependencies
Вся нейронная математика (matmul, relu, softmax, conv1d) — 
**~500 строк чистого Python**. Никакого numpy, torch, tensorflow.
Это возможно потому что модели МАЛЕНЬКИЕ (макс 10M параметров).
